🏭 案例一

智慧工廠報價助理

線材製造公司面對混亂詢價流程，如何用 AI 系統改善？學會分辨哪些環節該用 AI，哪些不該。

📖 問題說明

某線材製造公司每天收到 20–50 封詢價 Email 與 PDF，業務需要手動：
① 閱讀客戶需求 ② 搜尋歷史報價 ③ 確認庫存規格 ④ 手寫報價單

平均每封詢價需要 2–4 小時處理，且報價一致性差、錯誤率高。

🧠 系統拆解：哪裡用 AI？哪裡不用？

流程步驟 技術選擇 理由
讀取 Email / PDF，提取詢價欄位 LLM 非結構化文字 → 結構化資料，LLM 最擅長
查詢歷史相似報價案例 RAG 需從大量歷史資料中語意搜尋
查詢庫存、匯率、原料成本 API / MCP 即時數字，必須查 ERP 系統，不能靠 LLM 猜
套用標準價格公式計算報價 規則計算 精確計算，LLM 有幻覺風險，絕不能用！
生成報價單文件 LLM 格式化輸出，語言潤飾
業務主管最終審核 人工核簽 金額超過門檻，必須人工確認，AI 不能直接送出

🎨 案例二

AI 設計師助手

客戶需求模糊、設計師反覆溝通——如何用 AI 系統提升效率？

📖 問題說明

設計公司面對的痛點：
• 客戶說「我想要一個很酷的網站」，沒有具體需求
• 設計師花 60% 時間在需求釐清，而非設計本身
• 每次改版都要重新溝通，版本管理混亂

目標：用 AI 幫助設計師快速理解客戶需求、生成草稿、管理版本。

🧠 系統架構

功能 技術 說明
需求釐清對話 LLM + 對話流 引導客戶說清楚風格、目標族群、預算
搜尋設計參考 RAG 從公司設計庫中找相似風格案例
生成線框稿描述 LLM 產生 UI 元素說明與版面建議
版本記錄與比較 資料庫 + 規則 精確記錄，不能靠 LLM 記憶
客戶審核 & 簽核 人工流程 法律與商業責任，必須人工確認
❌ 常見錯誤：讓 LLM 直接生成「最終設計規格」並傳給工程師實作，沒有人工確認——結果方向跑掉，浪費大量工時。
✅ 正確做法：AI 負責「理解語言 + 搜尋參考 + 生成草稿」，人負責「判斷、決策、簽核」，系統負責「記錄、計算、執行」。

📚 案例三

AI 教學助教

出題、批改、解釋——AI 如何在教育場景中成為系統的一部分？

📖 問題說明

大學教授面對的挑戰：
• 200 人修課，作業批改耗費大量助教人力
• 同學在不同時間點需要個人化解釋
• 出題庫難以維護，容易重複

目標：用 AI 助教輔助出題、批改、個人化解釋，老師掌握最終評分。

🧠 功能模組 × 技術選型

功能 技術 AI 的角色
依教學目標出題 LLM 生成題目草稿，老師審核後入庫
搜尋題庫避免重複 RAG 語意搜尋，找相似題目
程式碼作業自動測試 自動化測試 不能靠 LLM，要跑測試案例
文字作業初步批改 LLM 給出評語與分數建議，老師確認
個人化錯誤解釋 LLM + Memory 根據學生的答題歷程，個人化說明
最終成績登錄 人工 + 系統 老師最終確認，AI 不能直接改成績

🔥 主線 Step 1

真實故事：混亂的詢價地獄

在設計解法之前，先進入問題的現場——感受業務人員每天在經歷什麼。

早上 8:30，業務小王打開電腦。

信箱裡有 23 封新郵件。其中 11 封是詢價——有日文、有英文、有夾著 PDF 的、有只寫「給我報個價」的。

他打開第一封，是日本客戶要 UL 認證的 18 AWG 線材，規格寫得很籠統。他得先去 SAP 系統查庫存，再去共用硬碟翻去年的報價紀錄，找到類似規格的案子，然後用 Excel 套公式計算....

到了下午 3 點，他才回覆完第 5 封。

今天還有 6 封在等。明天又會有 20 封進來。

而且上個月，他報錯了一個客戶的 MOQ，損失了一筆大訂單。因為他記錯了一個舊報價。

🔥 主線 Step 2

痛點分析

把模糊的「感覺很痛」拆解成可設計的結構化問題。

痛點 根本原因 後果
詢價格式不一致 客戶自行填寫，無標準化表單 業務需手動解讀，易誤解需求
歷史資料分散 報價存在個人電腦 / 共用硬碟，無統一系統 找不到相似案例，重複造輪子

🔥 主線 Step 3

設計思考 (Design Thinking)

用 5 個步驟，從「人的問題」出發，設計出真正有用的系統。

1
Empathize 同理
站在業務的立場思考

業務每天面對的不只是「找資料」，而是時間壓力 + 錯誤焦慮 + 系統切換的挫折感。他們真正想要的不是「AI 工具」，而是：

✦ 快速找到我要的資料
✦ 不用切換 5 個系統
✦ 報出去的價格不要出錯
✦ 不要加班

2
Define 定義問題
把觀察轉換成可設計的問題陳述
📌 問題陳述（HMW）：
「我們要如何幫助業務，在 30 分鐘內完成一封詢價回覆，且錯誤率低於 2%？」

注意：問題陳述要有可量化的目標（30 分鐘、2%），這樣才能在最後驗收時知道有沒有達到。

3
Ideate 發想功能
腦力激盪，不要太早收斂
📧 自動解析詢價 Email
🔍 歷史案例語意搜尋
📊 自動查詢 ERP 庫存
💰 公式計算報價初稿
📝 生成報價單文件
✅ 業務一鍵審核送出
4
Prototype 建 MVP
先做最小可行版本驗證假設

MVP 不等於完整系統，只要能驗證最核心的假設：

🎯 MVP 聚焦：只做「Email 解析 + 歷史案例搜尋」兩個功能，讓業務用 2 週，看看省了多少時間。其餘功能之後再加。
5
Test 驗證
用數據說話，而非感覺
驗收指標 目標值 測量方式
Email 解析準確率 > 90% 抽樣 100 封，人工核對
報價處理時間 < 30 分鐘 系統記錄時間戳
業務滿意度 NPS > 7 2 週後問卷
🔥 主線 Step 4（重點）

需求規格書 (Spec)

為什麼 AI 專案一定要先寫 Spec？因為 LLM 無法猜出你心裡想的系統。Spec 是人與 AI 協作的契約。

⚠️ 沒有 Spec 的後果：Prompt 改了又改、生出來的程式不符期望、每次換人問 LLM 都得到不同結果——這不是 AI 的問題，是你的問題。

📄 Spec 標準結構

1. 專案背景
說明為什麼要做這個系統、解決什麼問題、對誰有價值。
例：某線材公司每日處理 20+ 詢價，平均 3 天回覆，客戶流失率高...
2. 使用者角色 (Personas)
誰會用這個系統？他們的技術程度、使用習慣、最在乎的事？
例：業務小王（每天 20 封信，不擅長打字，最在乎速度和正確率）
3. 使用情境 (User Stories)
「身為 [使用者]，我想要 [功能]，以便 [達成目標]」
例：身為業務，我想要系統自動提取 Email 中的規格，以便不用手動整理
4. 功能需求 (FR)
系統「要做什麼」，每一條都要可測試
FR-01: 系統應能從 Email 提取：客戶名稱、產品型號、數量、交期
5. 非功能需求 (NFR)
效能、安全、可用性、資料隱私等
NFR-01: Email 解析時間 < 10 秒；NFR-02: 報價資料不得傳送到 LLM 供應商伺服器
6. 邊界 & 排除項目
明確說出不做什麼，防止範圍蔓延
本版本不支援：多語言（僅中英文）、自動送出報價（須人工審核）
7. 驗收標準
什麼叫做「做完」？可量化的定義
AC-01: 解析準確率 > 90%（抽樣 100 封）；AC-02: 業務完成報價時間 < 30 分鐘

🔥 主線 Step 7

實作流程：Prompt-Driven Development

現代 AI 系統開發，可以用 Prompt 驅動每個開發環節。學會這個流程，你就能讓 AI 幫你寫 AI 系統。

📄 Step ① 用 Prompt 寫 Spec
你是一位資深系統分析師。 請根據以下業務問題，生成一份完整的系統需求規格書（SRS），包含： 1. 專案背景 2. 使用者角色（Persona） 3. User Stories（每條格式：身為X，我想要Y，以便Z） 4. 功能需求（FR，每條可測試） 5. 非功能需求（NFR） 6. 系統邊界 7. 驗收標準 業務問題：{在此填入你的問題描述}
🧩 Step ② 用 Prompt 生模組設計
根據以下 Spec 中的功能需求，設計模組清單： - 列出所有需要的模組（Module） - 每個模組的職責（Responsibility） - 模組之間的依賴關係 - 建議的技術選型 Spec 節錄：{FR-01 到 FR-05...}

💡 生成後，你會得到一張模組地圖，之後每個模組可以獨立開發。

🔌 Step ③ 用 Prompt 生 API 設計
你是 API 設計專家。根據以下模組需求，設計 RESTful API： - 列出所有 endpoint - 每個 endpoint 的 Method、Path、Request Body、Response - 錯誤碼定義 - 認證方式 模組：Email 解析模組 功能：接收 Email 原文，回傳結構化需求資料

💡 有了 API 設計，前後端可以平行開發。

🗄️ Step ④ 用 Prompt 生資料 Schema
根據以下功能需求，設計資料庫 Schema（PostgreSQL）： - 定義所有 Table 結構 - 欄位名稱、型別、約束 - 索引建議 - 外鍵關係 需求：儲存報價單、歷史詢價、客戶資料

💡 AI 會幫你生成 CREATE TABLE SQL，你只需要 review 並微調。

🛠 Step ⑤ 用 Prompt 生程式骨架
用 Python + FastAPI 生成以下模組的程式骨架： 模組：EmailParserModule 功能： 1. 接收 email_text: str 2. 呼叫 OpenAI API 提取結構化欄位 3. 驗證必填欄位是否完整 4. 回傳 QuoteRequest 物件 請生成： - 資料模型（Pydantic） - Service 函式簽名（含型別標注） - 錯誤處理骨架 - 測試案例骨架（pytest）

💡 你不需要從零開始打，AI 給你骨架，你填入業務邏輯。

✅ Prompt-Driven 開發的核心：每個步驟都有清晰的輸入（Spec / 需求）和輸出（程式碼 / 文件），AI 負責生成草稿，人負責 Review 和決策。
📐 軟體工程觀念

SDD：規格驅動開發

Spec-Driven Development — 先寫規格，再寫系統。這是讓 AI 專案可控、可維護的基礎。

💬 為什麼 SDD 特別適合 AI 專案？

傳統軟體可以「先寫再改」，因為行為確定。但 LLM 的輸出是概率性的——沒有規格，你根本不知道 AI「做對了沒有」。

Spec 的三個功能：
① 溝通契約：讓 PM、工程師、業務對「做什麼」有共識
② Prompt 的起點：所有呼叫 LLM 的 Prompt 都應源自 Spec
③ 測試的依據：驗收標準 就是測試案例的基礎

🗺️ SDD 流程

📋
1. 撰寫需求規格書（SRS）
用 Prompt 生成草稿 → 業務確認 → 工程師 Review
↓
🧩
2. 從 Spec 推導系統設計
模組設計 → API 設計 → 資料 Schema
↓
💻
3. Prompt 驅動實作
每個模組的 Prompt 都對應到 FR 編號
↓
✅
4. 按 AC 驗收
每條驗收標準都有對應的測試案例
📌 報價助理對應：「AC-01: 解析準確率 > 90%」→ 生成 100 個測試 Email，自動跑解析，統計準確率。Spec 和測試直接連動。

TDD：測試驅動開發

Test-Driven Development — 先寫測試，再寫程式。這讓你在寫 AI 功能之前，就想清楚「正確」是什麼。

💡 TDD 的三個步驟（Red → Green → Refactor）
🔴
Red
先寫測試
測試失敗（因為還沒有程式）
→
🟢
Green
寫最少的程式
讓測試通過
→
🔵
Refactor
優化程式碼
測試要持續通過

🔴 測試一：缺漏欄位偵測
# test_email_parser.py def test_missing_product_code_raises_error(): """當 Email 中缺少產品型號時，應拋出 ValidationError""" email_text = "客戶要 100 條線，交期 2 週" # 缺少型號 with pytest.raises(ValidationError) as exc: parse_quote_email(email_text) assert "product_code" in str(exc.value) assert "缺漏欄位" in str(exc.value)
🔴 測試二：相似案例搜尋
# test_rag_search.py def test_similar_case_found(): """當輸入 18AWG UL 線材，應找到相似歷史案例""" query = QuoteRequest( product_type="wire", spec="18AWG", certification="UL" ) results = search_similar_cases(query, top_k=3) assert len(results) >= 1 assert results[0].similarity_score > 0.8 assert "18AWG" in results[0].description
💡 AI 專案 TDD 的挑戰：LLM 輸出是非確定性的，所以測試要關注「結構」和「約束」，而非「完全一致的字串」。例如：驗證回應是否包含必要欄位，而非逐字比對。

🎭 軟體工程觀念

BDD：行為驅動開發

Behavior-Driven Development — 用自然語言描述系統行為，讓業務和工程師說同一種話。

💡 BDD vs TDD 的差異
TDD BDD
寫給誰看 工程師 業務 + 工程師都能看懂
語言 程式碼（pytest） 自然語言（Gherkin）
關注點 函式的輸入輸出 使用者的行為情境
起點 FR（功能需求） User Story

📝 Gherkin 語法（報價助理範例）

Feature:
Email 詢價解析
Scenario:
不完整詢價應提示缺漏欄位
Given 客戶輸入了一封只有數量的詢價 Email
When 系統分析這封 Email
Then 系統應回傳「缺少產品型號」的提示
And 提示文字應以繁體中文呈現

Scenario:
完整詢價應生成報價初稿
Given 客戶提供了完整規格（型號、數量、交期、認證）
When 系統處理這封詢價
Then 系統應生成包含單價與總價的報價初稿
And 業務應能在介面上看到並編輯這份初稿

Scenario:
金額超過授權上限需主管審核
Given 報價金額超過 NT$ 500,000
When 業務準備送出報價
Then 系統應自動發送審核通知給業務主管
And 報價單狀態變更為「待審核」

🔗 BDD 在 AI 專案的價值

📢 業務語言 ↔ 技術規格

Gherkin 讓業務和工程師能確認「做的是不是同一件事」，避免做完才發現方向錯誤。

🤖 Prompt 的驗收框架

每個 Scenario 都是一個 LLM 功能的驗收情境，可以自動化測試 AI 的行為。

🎯
最終任務：用 Prompt 生成 BDD 情境

把你的 User Story 輸入右側，讓 AI 生成對應的 Gherkin 情境描述。
